## 1. 执行摘要

### 1.1 发布结论

**结论：不建议按当前提交直接发布 1.2.0。**

项目的基础工程质量不错：默认 Debug/Release、静态/共享、关闭 async、关闭 crash、最小功能、unlimited line、静态/动态 zlib 等主流构建均能完成；全新目录中的现有测试全部通过；Windows DLL 的 51 个公开符号在默认、no-crash、minimal 三种构建中完全一致；安装后的 C/C++ 消费项目也能正常编译、链接和运行。

但深度验证确认了 3 个严重问题：

1. 轮转清理会把符合“相同前缀和扩展名”的普通文件误认为备份并删除，存在用户数据损失风险。
2. async 模式并发执行 `hx_clog_reconfigure()` 会静默丢日志；即使 overflow policy 为 `BLOCK`，丢失也不会进入 `dropped_lines`。
3. 首次初始化之前调用公开 API `hx_clog_get_stats()` 会在 Windows 触发访问冲突，而不是返回错误。

按本次约定的发布门槛，只要存在未解决的数据丢失或公开 API 崩溃类严重问题，就应判为“不建议发布”。

## 5. 严重发现

### S-01 轮转清理会删除非轮转文件

- **位置**：[`src/hx_clog_rotate.c:201`](../../src/hx_clog_rotate.c#L201)、[`src/hx_clog_rotate.c:438`](../../src/hx_clog_rotate.c#L438)
- **触发条件**：同一日志目录存在文件名以 `<stem>.` 开头并以目标扩展名结尾，例如 active 为 `audit.log`，普通文件为 `audit.notes.log`；同时启用数量保留清理。
- **根因**：候选收集只验证前缀和后缀，没有要求文件名完整匹配 `<stem>.YYYY-MM-DD.<index>.<ext>[.gz]`。解析失败时仍保留候选，并用 mtime 作为排序 key。
- **动态证据**：最小复现先创建 `audit.notes.log`，再触发 size rotation；完成后输出 `unrelated-file-exists=0`。
- **影响**：库可删除或压缩应用放在日志目录中的非轮转文件，属于明确的数据损失风险。
- **建议**：收集阶段必须完整解析日期、正整数 index、扩展名和可选 `.gz`，解析失败立即忽略；增加 sentinel 文件回归测试，并覆盖无扩展名 active file。

### S-02 async reconfigure 静默丢日志且统计为 0

- **位置**：[`src/hx_clog_core.c:1345`](../../src/hx_clog_core.c#L1345)、[`src/hx_clog_core.c:1752`](../../src/hx_clog_core.c#L1752)、[`src/hx_clog_async.c:140`](../../src/hx_clog_async.c#L140)
- **触发条件**：async + `HX_CLOG_OVERFLOW_BLOCK`，生产线程持续写入，另一线程调用 `hx_clog_reconfigure()`。
- **根因**：reconfigure 先停止 async engine，但在新 engine 启动前全局 mode 仍是 async；这段窗口中的日志进入 `hx_async_enqueue()`，因 `running/stop` 返回错误。`core_dispatch_line()` 忽略返回值，该路径也没有增加 dropped counter。
- **动态证据**：两次独立运行各尝试 200000 行：
  - delivered 183682，丢失 16318，`reported_dropped=0`；
  - delivered 183402，丢失 16598，`reported_dropped=0`。
- **影响**：运行时配置更新会无告警破坏 BLOCK 的“不丢失”语义，监控统计还会显示零丢失。
- **建议**：在停止旧队列前建立写入过渡协议，例如先原子切换到同步路径或阻塞生产者，再 drain/swap/restart；所有 enqueue 失败必须计入 dropped 并触发节流错误通知；增加并发 reconfigure 测试，验收条件为 `attempted == delivered` 或明确、可统计的策略化丢弃。

### S-03 初始化前调用 `hx_clog_get_stats()` 访问冲突

- **位置**：[`src/hx_clog_core.c:1429`](../../src/hx_clog_core.c#L1429)
- **触发条件**：进程首次使用 hx_clog 时直接调用 `hx_clog_get_stats(&stats)`。
- **根因**：函数没有调用 `ensure_once()`，直接锁定尚未初始化的 `g_core.stats_lock`。同文件的 `hx_clog_get_sink_count()` 已正确调用 `ensure_once()`，行为不一致。
- **动态证据**：独立子进程退出码 `-1073741819`，即 Windows `0xC0000005` access violation。
- **影响**：合法非空参数即可使进程崩溃；该 getter 返回 `int`，调用方合理预期错误码而不是未定义行为。
- **建议**：入口先执行 `ensure_once()`，再明确选择返回全零统计或 `HX_CLOG_ERR_NOT_INITIALIZED`；新增 before-init、after-shutdown、并发读取测试。

## 6. 中等发现

### M-01 POSIX after-fork 通过重复 `pthread_mutex_init` 重建锁

- **位置**：[`src/hx_clog_core.c:1642`](../../src/hx_clog_core.c#L1642)、[`src/hx_clog_file.c:203`](../../src/hx_clog_file.c#L203)、[`src/hx_clog_async.c:388`](../../src/hx_clog_async.c#L388)
- **问题**：child helper 对已经初始化、可能处于 locked 状态的 mutex/cond 原地再次调用 `pthread_mutex_init/pthread_cond_init`，且忽略返回值。对已初始化同步对象重复初始化不具备可移植定义，恰好可能使该 API 无法解除 fork 时继承的锁状态。
- **建议**：使用 `pthread_atfork` 协议或经验证的 child-side 重建策略；增加 Linux sync/async 多线程 fork 集成测试，验证 child 可以继续 write/flush/shutdown。

### M-02 POSIX callback sink 重入会自锁

- **位置**：[`src/hx_clog_core.c:885`](../../src/hx_clog_core.c#L885)、[`src/hx_clog_sink.c:200`](../../src/hx_clog_sink.c#L200)
- **问题**：用户 callback 在 `g_core.sink_lock` 持有期间执行。Windows `CRITICAL_SECTION` 是递归锁，本地 `get_sink_count()` 重入成功；POSIX 默认 pthread mutex 非递归，同样回调调用 `flush/get_sink_count/remove_sink` 会死锁。callback 文档没有禁止重入。
- **建议**：在锁内建立稳定 sink 快照，在锁外调用用户代码，或显式规定不可重入并增加检测；补充 POSIX 回归测试。

### M-03 运行时统计与 duplicate 开关存在普通变量数据竞争

- **位置**：[`src/hx_clog_async.c:413`](../../src/hx_clog_async.c#L413)、[`src/hx_clog_core.c:1437`](../../src/hx_clog_core.c#L1437)、[`src/hx_clog_core.c:1590`](../../src/hx_clog_core.c#L1590)、[`src/hx_clog_core.c:2041`](../../src/hx_clog_core.c#L2041)
- **问题**：async counters 在 queue lock 下写、无锁读取；`g_dup.enabled` 在 mutex 下写、hot path 无锁读。当前 TSan job 没有并发调用这些控制/查询 API，因此没有覆盖到该组合。
- **建议**：使用原子变量或在一致锁下快照；新增持续 write 同时 get_stats/toggle suppression 的 TSan 测试。

### M-04 长 context key 会重复占用 16 个槽位

- **位置**：[`src/hx_clog_core.c:575`](../../src/hx_clog_core.c#L575)、[`src/hx_clog_core.c:615`](../../src/hx_clog_core.c#L615)
- **问题**：key 保存时截断到 63 字节，但查找时用原始长 key 与截断值比较；同一长 key 每次 put 都被当成新 key，最终容量耗尽，remove 也找不到原始 key。
- **动态证据**：连续 17 次 put 同一个 95 字节 key，最后返回 `-1`，输出 context 含多份相同截断 key，长度 990 字节。
- **建议**：入口先规范化/验证长度；可选择明确拒绝超长 key，或以相同截断/哈希规则查找。文档应公开 key/value/item 上限。

### M-05 重复 `%v` pattern 会远早于行大小上限截断

- **位置**：[`src/hx_clog_core.c:1725`](../../src/hx_clog_core.c#L1725)
- **问题**：heap retry 容量只按 `message_len + context_len + 2048` 估算，没有考虑 pattern 中 `%v/%x/%F` 的重复次数。formatter 返回的又是“已写长度”而非 required length，因此无法再次扩容。
- **动态证据**：100 个 `%v`、10000 字节消息，理论展开 1000000 字节、默认 cap 524288 字节，实际只交付 12047 字节。
- **建议**：format pass 返回 required length，或先做精确 sizing pass；随后应用统一 cap 并二次格式化。

### M-06 轮转清理最多处理 512 个候选

- **位置**：[`src/hx_clog_rotate.c:343`](../../src/hx_clog_rotate.c#L343)、[`src/hx_clog_rotate.c:381`](../../src/hx_clog_rotate.c#L381)
- **问题**：超过 `HX_ROTATE_MAX_BACKUPS` 后其余文件被静默忽略，`max_backup_days/max_backup_files/max_compressed_files` 无法保证最终上限。默认 compressed retention 为 unlimited，长期进程很容易超过 512。
- **建议**：使用动态增长数组、流式 top-N，或按批次完整遍历；增加 600+ archive 测试。

### M-07 POSIX Crash 路径的 signal-safety 与安装结果是 best-effort

- **位置**：[`src/hx_clog_crash.c:115`](../../src/hx_clog_crash.c#L115)、[`src/hx_clog_crash.c:584`](../../src/hx_clog_crash.c#L584)、[`src/hx_clog_crash.c:634`](../../src/hx_clog_crash.c#L634)
- **问题**：handler 内仍调用 `snprintf/localtime_r/backtrace/_Unwind_Backtrace/dladdr` 等非严格 async-signal-safe 操作；ring dump 与正常写入之间也没有同步。安装函数忽略 mkdir、sigaltstack、sigaction 的失败，却返回 OK。
- **建议**：文档明确为 best-effort；预计算目录/固定前缀，检查所有安装调用返回值；增加 POSIX 真实 SIGSEGV、stack overflow、安装失败测试。

### M-08 MinGW 并发格式化使用非线程安全 `localtime()`

- **位置**：[`src/hx_clog_core.c:304`](../../src/hx_clog_core.c#L304)
- **问题**：MinGW 分支调用共享静态缓冲的 `localtime()` 后复制，多个业务线程可并发进入；README 列出 MinGW 构建方式，但 CI 的 Windows job 仅覆盖 MSVC。
- **建议**：使用 MinGW 可用的线程安全时间转换或加专用锁；增加 MinGW CI 和并发 timestamp 测试。

### L-02 rotation/date/path 测试不清理工作目录

- fresh zlib static 10/10 通过；复用旧 `build_zlib` 时 `test_rotate` 因跨日期残留索引失败。
- 多个 path/date 测试只检查文件“存在”或数量下限，历史产物可能掩盖本次失败。
- 每个测试应使用唯一临时目录并在开始/结束清理；不应依赖历史 archive 索引连续。

### L-03 公开 API 文档与直接测试仍有空洞

- 5 个 API 没有名称级文档；platform sinks、fork、reopen、crash callback、多个 getter/`va_list` 入口没有直接测试。
- 建议从头文件自动生成 API inventory，并在审查/CI 中检查 declaration → docs → test 映射。

### L-04 Unix shared library 缺少显式 ABI/可见性策略

- **位置**：[`CMakeLists.txt:70`](../../CMakeLists.txt#L70)
- target 未设置 `VERSION/SOVERSION`，也未设置 hidden visibility；Unix 默认可能导出内部全局符号。静态 target 也没有显式 PIC 选项。
- 建议为正式共享库建立 SOVERSION、默认 hidden + `HX_CLOG_API` 导出策略，并验证 static-to-shared consumer。

### L-05 CI 的 packaging smoke 只检查文件存在

- **位置**：[`.github/workflows/ci.yml:131`](../../.github/workflows/ci.yml#L131)
- 当前 job 不执行 `find_package` consumer。此次本地静态/共享 C/C++ 消费均成功，说明现状可用，但回归保护不足。
- 建议把本次 consumer smoke 纳入 CI，并覆盖 zlib static dependency。

### L-06 缺少正式发布材料与自动化

- Git 仓库和 GitHub API 均显示 0 tag、0 Release；仓库没有 CHANGELOG、SECURITY、CONTRIBUTING 或 release workflow。
- 对公开第三方库，建议至少增加 CHANGELOG、支持/安全策略、版本 tag、源码包和最小 release checklist。

## 8. 信息项

### I-01 配置文件加载与 benchmark 是明确的路线图缺口

README 将配置文件加载描述为“可选扩展”，将 benchmark 列为推荐指标和 Phase 4 项目。它们没有被当前 API 文档宣称为已交付能力，因此本报告不将其计为功能缺陷；但没有 benchmark 意味着吞吐、延迟和内存声明无法形成版本基线。

### I-02 C++ wrapper 是“薄封装”，并非 C API 全覆盖

当前 wrapper 足以满足基础 RAII 与便捷格式化，但没有拥有式封装 `system_logger_name`、custom formatter 等全部 C 配置项。建议明确其定位为 convenience subset，或补齐 setter/字符串生命周期管理。

### I-03 MSVC `/analyze` 告警已人工复核

在关闭 MSVC CRT deprecation 噪声后，`/W4 /WX` 通过；`/analyze` 给出 6 个告警，涉及数字反向缓冲、`split_name`、rotation 临时字符串和 async slot。结合控制流和已有 NUL/分配不变量，本轮未确认这些告警为真实越界，但建议在代码中改写或增加断言以消除分析歧义。

## 10. 整改优先级与验收标准

### P0：发布前必须完成

1. 严格解析轮转文件名，保证普通文件永不进入 cleanup；加入 sentinel 回归测试。
2. 修复 async reconfigure 过渡期，BLOCK 场景不得静默丢失；任何策略化丢失必须进入 stats/error handler。
3. 修复 `hx_clog_get_stats()` 首次调用崩溃，并覆盖全部公开 getter 的 before-init/after-shutdown 行为。

### P1：建议 1.2.0 一并完成

1. 重做或验证 POSIX after-fork 锁重建协议。
2. 解决 callback 重入锁问题，或形成清晰且可检测的不可重入契约。
3. 消除 async stats、duplicate 开关及 reconfigure 热路径的数据竞争。
4. 明确 context 长度策略，修复长 key 重复插入。
5. 改为精确计算格式化所需容量，保证只在配置 cap 处截断。
6. 让 retention 覆盖 512+ archive，并增加相关压力测试。

### P2：发布工程完善

1. 修复自定义 line cap 测试和所有运行目录隔离问题。
2. 加入 MinGW、真实 fork、真实 POSIX crash、system sink 和 package consumer CI。
3. 补齐 API 文档、SOVERSION/visibility、CHANGELOG、SECURITY、tag 与 Release 流程。
4. 建立最小性能基线：sync/async 单多线程吞吐、enqueue p50/p99、drop 行为和峰值内存。

### 发布复验门槛

- P0 的三个最小复现全部转绿；
- 默认与 options matrix 全部通过；
- ASan/UBSan/TSan 目标化覆盖 reconfigure/get_stats/duplicate 场景；
- static/shared、zlib on/off、安装消费与 51-symbol ABI 检查继续通过；
- fresh 与重复运行测试结果一致，不依赖工作目录历史文件。

## 11. 最终判断

`hx_clog` 已具备一个小型跨平台日志库的大部分主体能力，C ABI、功能门控、安装消费和 CI 平台矩阵尤其扎实。当前阻碍 1.2.0 发布的不是“功能数量不够”，而是三个位于异常/组合路径的可靠性问题：轮转误删、异步重配置无统计丢失、初始化前 stats 崩溃。先解决这三个问题，再处理并发与测试隔离缺口，项目即可从“功能丰富但有发布风险”进入“可条件发布”的状态。
